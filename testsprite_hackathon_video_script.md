# Draftr x TestSprite Hackathon Video Script

This version is written as a fuller 3 to 3.5 minute narration and follows the actual testing sequence instead of jumping straight to the final score.

---

For this hackathon, we used TestSprite as a serious product-testing partner for Draftr, our app that turns URLs, PDFs, and notes into short-form videos. What made it useful was not just the pass rate. It was the way it helped us move from a rough local build, to a stable local build, to a production-validated product, while also separating real bugs from test-environment limitations.

We started with the website in local mode, and our first meaningful baseline was 6 out of 15 tests passing. At that point, the app idea was already visible. The home page, the chat entry point, and some basic flows were there. But TestSprite showed us that the experience was still fragile in exactly the places that matter for a real product: navigation between pages, guest access into the shared shorts library, state transitions around login and logout, and the reliability of loading shorts in the feed. That first run was important because it gave us our map. Instead of saying "the frontend feels off," we knew which workflows were actually weak.

After that, we fixed the first layer of UX and flow issues and ran the website locally again. That second local run improved to 8 out of 15. At that stage, TestSprite confirmed that a lot of the foundation was working. It could create and load chats correctly, the main home-to-chat journey still worked, the login page was reachable, the authenticated profile experience was working, the metadata panel worked, and even the guest flow from login into the general library was working in that run. So the app was clearly moving in the right direction. But TestSprite also showed us what was still not production ready. The navbar still had discoverability problems around profile access, direct browsing of the general library still had failures, opening shorts in a new tab was not reliable yet, playback and mute controls were not consistently testable, and a few interaction-heavy tests were still blocked. That second run was the point where we knew the core app existed, but the edge cases were still real.

Then came the third local website run, and this is the one we treated as the final local gate. That run reached 11 out of 15, which was a 20 point jump over the previous local run. More importantly, the exact kinds of issues we cared about were getting resolved. TestSprite confirmed that core navigation improved, direct guest browsing of the shorts library improved, feed navigation improved, opening a short in a new tab improved, and the logout flow was now working end to end. So this was no longer just a nice-looking prototype. It was becoming a stable product.

At the same time, the third run also told us why it was the right moment to stop doing only local checks and move to deployed validation. The remaining issues fell into three buckets. First, there was one design expectation gap: TestSprite wanted a profile link visible from the navbar even for guests, while our navbar only showed that link for authenticated users. Second, there was one genuine regression around continuing as guest from the login page into the shorts view. Third, there were issues that looked more like environment and timing constraints than clean product bugs, like intermittent sidebar loading and MP4 playback through the tunnel. So the third run was "final" not because everything was perfect, but because it had reduced the problem space. Most remaining questions could only be answered by testing the deployed app.

So we deployed and ran the full production website suite. That was the strongest frontend result: 21 out of 24 tests passed, or 87.5 percent. The guest flow from login was fixed in production. The shorts sidebar loaded consistently. Authentication, logout, and profile flows all passed. The marketing pages passed. The empty-prompt guardrails passed. The video count stepper boundaries passed. That production run also helped us isolate what was left. One remaining issue was a real UI bug: the like button changed internal state, but the heart icon did not change visually enough. The other big remaining issue was MP4 playback through the TestSprite tunnel, and that was valuable because it told us not to waste time "fixing" something that was really a test-environment media constraint.

We also used TestSprite to test backend modules separately, because we did not want to rely only on polished end-to-end demos. The recommendation system had a very real testing journey too. On the first attempt, it was effectively 0 out of 7 because the previous run had left the config in a bad state, so TestSprite returned empty code and the suite failed before real execution even started. We reset the setup, retried the run, and then the recommendation system came back at 5 out of 7. What made that useful was that the remaining failures were not backend bugs. They came from generated test code using `response.json()["chat_id"]` instead of the correct nested response shape, `response.json()["chat"]["id"]`. So even there, TestSprite helped us separate actual system problems from test-generation assumptions.

The video generator backend told an even stronger story. The first run passed only 1 out of 10, but that first run was incredibly valuable because it surfaced a real bug: invalid non-UUID path parameters were returning 500 instead of 422. It also exposed several places where the generated tests misunderstood the API contract, like wrong paths, wrong content types, and missing multipart form expectations. We fixed the UUID validation issue, tightened route behavior, clarified the contract, and reran the suite. The result jumped to 8 out of 10. That was a huge signal that TestSprite was doing real work for us, because it pushed us past superficial success and into contract-level correctness.

After that, we ran the full backend suite, and it reached 13 out of 15. At that point, the backend passed health, batches, assets, SSE streaming, agents, chats, engagement, recommendations, and video-edit coverage. The remaining failures were again TestSprite code-generation issues, not backend correctness issues. So by the end of this process, we were not just saying "the backend seems okay." We had strong evidence that the backend was stable across multiple modules and multiple test plans.

One strong limitation we faced was consistency in generated test code, especially around nested response shapes like `chat.id`. To make that easy to review, we also generated a separate issues report so the TestSprite team can clearly see where the test generator helped us, and where it still needs improvement.

Across all of these repeated local, production, and backend runs, we pushed through roughly 150 test scenarios and used them to improve real parts of the product. The local website journey moved from 6 out of 15, to 8 out of 15, to 11 out of 15. The deployed website reached 21 out of 24. The video generator backend improved from 1 out of 10 to 8 out of 10. And the full backend closed at 13 out of 15. That translated into real improvements in guest mode, auth and logout flows, shorts loading, sidebar reliability, open-in-new-tab behavior, UUID validation, backend contract correctness, and overall release confidence for Draftr.
